# Written Response

## 4.1 Approach

The intuition I started with was simple: a Pydantic schema in the middle, an LLM filling it from the PDFs on one side, and authoritative external sources validating the structured output on the other. The schema is the contract; everything else is replaceable.

**Why a schema as the contract.** Heterogeneous Dutch zoning packets, three document types per project, dozens of clauses per document. Without a single structural contract the pipeline becomes a tangle of stage-specific shapes and reviewers cannot trust any individual value. With one Pydantic model in the middle, every extracted value must arrive with `Provenance` (source document, page, verbatim Dutch quote) and `Confidence` (0–1 score plus named flags). A value with no provenance cannot exist in the framework. This is the Scenario 1 defence at the schema level: silent errors are structurally impossible because the schema refuses to hold them.

**Why a layered reliability stack rather than one extraction pass.** Each layer catches a different failure mode. A Sonnet 4.5 multimodal pass reads each page image plus text layer against the schema, with Dutch planning vocabulary supplied as glossary context (seeded from the Stelselcatalogus API, with a curated fallback path when the API key is missing). A second independent pass (`critical_fields.md`) re-asks the same questions on heights, setbacks and parking without seeing the first pass; disagreements lower confidence and surface in the viewer. A reconciliation step (`reconcile.py`) compares regels height clauses against the kaveltekening's spatial-proximity readings and overrides drawing-derived values when the regels speaks directly. An IMRO Ruimtelijke Plannen API layer (`cross_validate.py`, automated in the pipeline) diffs every `NumericalConstraint` against the authoritative published plan within a 5% tolerance, populating `cross_validation.agreement` on each constraint and adding `imro_api_disagreement` to its flags. Universal sanity bounds catch order-of-magnitude impossibilities. A human reviewer gates the final output. None of these layers is sufficient alone; the combination is what makes a confidently-wrong 32 m implausible.

**Why a second authoritative source in parallel (the GML route).** Late in the build I got the Ruimtelijke Plannen API key and discovered I could fetch the CityGML representation of the same site directly from PDOK. The GML is designed for machines: every `Functieaanduiding` carries an explicit `xlink:href` linking it to its `Bouwvlak`, no spatial guessing required. This solves the deepest pain of PDF parsing on a kaveltekening, where overlapping polygon layers (bouwvlakken, functieaanduidingen, bouwaanduidingen) are drawn on top of each other for human reading and the parser has to guess which label binds to which polygon. The GML approach is wired into the pipeline as a parallel zone framework (`data/outputs/<project>/approach_gml/`) and into the Streamlit viewer as a side-by-side comparison page ("Approach 2 — GML"). What is not yet automated is a per-NumericalConstraint diff between LLM extraction and GML-derived values inside the main framework — that diff is currently visual rather than enforced. I take the architectural point rather than just the demo result: this is how Scenario 1 should be answered at scale. The authoritative GML and IMRO API are the cross-source ground truths; in production every numerical constraint and every zone polygon would be diffed against them automatically and a disagreement would block sign-off rather than being a UI flag.

For the Draka packet specifically the IMRO API endpoint was unavailable during the run (recorded as `agreement='not_attempted'` per the graceful-degradation contract), so the GML route ended up doing the heavy validation lifting in practice. That outcome is itself part of the failure-honesty story: the layered defence kept working even when one layer fell over.

**Why no AI in some places.** Scale-factor detection from the kaveltekening reads the PDF Measure dictionary or parses "Schaal 1:N" from the embedded text. Label classification on the verbeelding (ALL-CAPS = bestemming, parens = functieaanduiding, brackets = bouwaanduiding) is pattern matching. GML parsing is `lxml` and `shapely`. The rule was: use the LLM for interpretation of ambiguous natural language, not for structured lookups. A scale annotation does not need interpretation; a clause about what constitutes a "plint" does.

**What worked, what hurt.** The schema-as-contract decision paid off immediately. Modules could be rewritten in isolation as long as they produced or consumed valid `ParametricFramework` instances. The hardest part by a wide margin was the kaveltekening parser. Vector PDFs gave clean deterministic output for plot outlines and labels, but the overlapping-polygon-layer problem and the various ways spatial proximity fails (zones drawn slightly larger than the bouwvlakken they cover, height-marker diamonds that steal IMRO codes) consumed disproportionate time. Discovering the GML route afterwards felt slightly painful for that reason: had I started there, half the geometry-parsing complexity would have evaporated.

The geometry parser uses deterministic vector extraction because the input is a CAD-exported vector PDF where polygons and labels are structured data, not pixels. This is faster, cheaper, and more accurate than OCR for this class of input. The parser explicitly returns `status='manual_input_required'` with a populated `reason` when it encounters PDFs without extractable vectors (scanned drawings, photographed plot proposals, raster-only exports). A one-week extension would route those cases through a multimodal LLM vision fallback that produces the same `Geometry` model from rendered page images, broadening coverage to older bestemmingsplannen and small-municipality submissions where vector PDFs are not available.

A practical detail worth flagging: the parallel multimodal extraction initially hit Anthropic's 30,000 input-tokens-per-minute rate limit hard enough to fail on most pages. The fix is in `extract.py`: an `asyncio.Semaphore(1)` constrains per-page concurrency, a 429-detecting retry loop with exponential backoff handles transient throttling, and a 5-second inter-page delay paces the retry-of-failed-pages path. Slowing down the pipeline was cheaper than upgrading the API tier for a prototype.

The pipeline runs end-to-end on Draka in roughly 2 hours on a cold run, dominated by extraction API calls. Subsequent runs detect cached expensive outputs (`extraction_raw.json`, `geo_context.json`, `programme.json`, `imro_cross_validation.json`) and re-run only the cheap downstream chain in seconds, controllable via `--force` and per-stage `--skip-*` flags.

## 4.2 AI Choices

**Where I used the LLM.** Two places: structured extraction (Sonnet 4.5, page-by-page, schema-constrained, with the Stelselcatalogus glossary injected when relevant terms appear) and programme inference (Opus 4.7, with a reasoning-trace requirement that cites primary evidence per decision). Both are constrained to Pydantic output via PydanticAI, so the model cannot return free-form prose. Both require every numerical claim to cite either verbatim source text or a named designer judgment. These constraints do most of the reliability work; the model selection matters less than the contract it has to satisfy.

**Where I deliberately did not use the LLM.** Deterministic lookups stay deterministic: scale detection, label-pattern classification, GML parsing, IMRO API cross-validation, sanity-bounds checking. Anywhere a regex or a typed parser produces a stable answer, that is the right tool. LLM variance on those tasks is pure cost and zero benefit.

**On honesty in confidence scores.** The hardest LLM-engineering decision was forcing the model to produce its own confidence numbers with reasons. It will happily output 0.95 on everything if you let it. The prompts explicitly forbid that and ground confidence in named conditions ("0.85 to 0.99 for clear statements with minor ambiguity, 0.5 to 0.85 for values requiring interpretation or with hedging language"). The output is still imperfect; some confidently-wrong values slip through. That is exactly why the external authoritative sources matter and why I would not ship without the GML diff promoted to a hard blocker.

## 4.3 What I Would Do With Another Week

**Most valuable: enforce the GML diff as a constraint-level gate, not a visual one.** The GML zone framework is already produced per project. The remaining step is to walk every `NumericalConstraint` and every bouwvlak polygon in the assembled framework, diff them against the GML equivalent, populate `cross_validation` on each, and reject sign-off when any disagreement exceeds tolerance. About one day if I generalise the GML zone-code matcher beyond Draka's codelists, which is the bottleneck. This closes the largest remaining gap in the Scenario 1 story for any plan where GML is available.

**Second: a Grasshopper round-trip with a real GH engineer.** The framework outputs GeoJSON, COMPAS meshes, OBJ massings, and a flat parameters JSON. I designed those by reading the OMRT stack, not by talking to anyone who actually builds GH definitions. One afternoon with a real engineer would tell me whether the coordinate system, the schema shape, and polygon orientation are usable as-is or need rework. That feedback is worth more than any feature.

**Third: DSO Omgevingsdocumenten cross-validation.** Post-2024 omgevingsplannen use a different endpoint with a slightly different schema. The architecture already supports it (the `cross_validation` model is source-agnostic), so this is a one-day extension.

**What I would cut.** The dual-pass diff view in the viewer is overbuilt; the IMRO/GML cross-checks are stronger and cleaner disagreement signals. I would keep the dual-pass as a confidence modifier but remove its dedicated UI surface.

## 4.4 Productisation

**What is already production-shaped.** The schema, the layered reliability stack, the IMRO cross-validation, the smart-skip CLI (`omrt run <input_dir>` detects cached expensive outputs and re-runs only the cheap downstream chain, with explicit per-stage skip flags), the graceful-degradation contract on every external API, and `scripts/sanity_check.py` which validates any project's output directory against structural contracts. That last one becomes a CI gate between extraction and the PM review queue in production.

**What is not.** The Streamlit viewer is a demo surface, not a daily-use tool. The pipeline runs synchronously in a terminal, not as a background job. There is no access control, no audit trail of overrides, no integration with OMRT's existing intake flow. In production a PM uploads through the existing intake interface, a job queue runs the pipeline asynchronously, a proper web app replaces the viewer with authenticated per-field overrides logged to a database, and the approved framework flows directly into Run via an API rather than being handed off as a JSON file. The naive JSON-on-disk archive (`data/archive/projects/`) becomes a real database — Postgres for the schema and a vector index for the few-shot retrieval the knowledge layer currently stubs.

**Where the human stays.** Programme inference review is always human. Every red flag (IMRO disagreement, GML disagreement, low confidence, outside sanity bounds, dual-pass disagreement) requires explicit resolution; the system never auto-resolves. The final send-to-Grasshopper gate is always human.

**What breaks at scale.** IMRO/DSO API fair-use limits during busy intake periods (mitigation: per-municipality snapshots refreshed nightly rather than per-query calls). LLM extraction costs scale linearly with page count, so a 200-page toelichting costs significantly more than Draka (mitigation: cheaper first-pass model for triage, Sonnet/Opus only on flagged regions). Schema migrations become painful when older projects in the archive no longer validate; the `verification_status` field acts as a version-pin and schema versioning becomes mandatory past a few dozen projects. Naive `glossary.json` lookups become slow once the catalog grows beyond the Stelselcatalogus seed.

**Which teams I would need.** Backend engineer for the queue and storage migration; small UX investment for the production viewer; one Grasshopper engineer's time for schema refinement; a PM champion who tests every release on real intake packets and gives honest feedback about where it fails.

## 5. Scenario Reflections

### Scenario 1: The confident hallucination

This is exactly what the layered defence is built for, and exactly what the second-source approach answers most cleanly. For any Dutch project with a published IMRO plan ID, every numerical constraint is already cross-validated against the Ruimtelijke Plannen API in the pipeline today (`cross_validate.py`), and a parallel GML route gives a second authoritative comparison on zones and geometries. A 32 m extracted value against a 23 m authoritative value falls far outside any reasonable tolerance; the disagreement gets flagged, confidence drops by 0.3, both values surface side by side in the viewer with the regels clause quoted verbatim, and the PM cannot mark the framework as reviewed without resolving the flag. The remaining gap in the current prototype is that the GML diff is visual rather than enforced at the constraint level; promoting it to a hard gate is the single most important next step (Section 4.3).

For projects without a plan ID, or when the API is unreachable (as happened on the Draka run itself, recorded as `agreement='not_attempted'`), the remaining layers still operate: dual-pass extraction, regels-vs-toelichting cross-document checking, regels-overrides-verbeelding reconciliation, universal sanity bounds, per-field confidence flagged below 0.85. A 32 m hallucination would have to pass all of them simultaneously, including the independent dual pass producing the same wrong value, which is possible but unlikely. The roles are clear: the model produces extraction with explicit uncertainty, the product cross-validates and surfaces flags (and in production, blocks sign-off on disagreement), the human resolves them. The system cannot guarantee zero errors. It can guarantee that errors are visible and that nothing pretends to be ground truth.

### Scenario 2: The next prototype

The natural next move is to turn the archive of structured outputs into something OMRT can actually retrieve against. Every framework this pipeline produces is already a normalised, schema-validated representation of a project. Treated as an archive, the collection becomes a searchable knowledge base of past intakes: similar sites, comparable programme briefs, what constraint patterns recurred, what KPI trade-offs the Run revealed. A PM starting a new intake could surface relevant past projects automatically rather than relying on team memory.

The leverage comes from doing this on structured data rather than free text. Because every value has provenance, every constraint has a category, and every project has a comparable schema shape, the matching can be done on real attributes rather than vector embeddings of prose. The seven-day prototype would build the retrieval surface (most-similar-project given a new intake's constraints and location) on top of three to five seeded archived projects beyond Draka, and validate it against past projects with known reuse stories. Worth keeping if the retrievals actually shorten intake time. Worth killing if PMs end up scrolling past every suggestion because the similarity is too shallow.
